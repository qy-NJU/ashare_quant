# A股量化数据流重构与优化计划

## 一、 当前数据获取与存储的痛点分析

在目前的 `ashare_quant` 框架中，数据获取和存储存在以下严重影响效率和稳定性的问题：

1. **“On-the-fly”（即时）下载模式**：
   - 目前的逻辑是：回测或训练执行到某只股票的某个时间段时，才去触发数据下载。这导致网络请求与核心计算流程强耦合。一旦 Baostock/AkShare 接口限流、网络波动或服务端抽风，整个长达数小时的训练任务就会直接崩溃。
2. **碎片化的缓存设计 (Cache Miss 严重)**：
   - `repository.py` 中的缓存键设计为 `f"{symbol}_{start_date}_{end_date}_{adjust}.parquet"`。
   - 这意味着，如果你今天跑了 `2023-01-01` 到 `2023-06-30` 的数据，明天想跑 `2023-01-01` 到 `2023-07-01`（仅仅差了一天），系统会认为缓存未命中，从而**重新下载整段数据**。这种设计使得缓存形同虚设，极大地拖慢了研发效率。
3. **特征重复计算**：
   - 每次获取到 K 线数据后，都会重新跑一遍 `FeaturePipeline`。对于 130+ 个技术指标来说，这也是巨大的性能开销。

## 二、 稳定的数据源替代方案

免费的 API（如 Baostock, AkShare）天然存在限频和不稳定的问题。为了解决数据源问题，建议采用以下分层方案：

### 1. 数据源选择
- **首选 (性价比最高)：Tushare Pro**
  - **优势**：业界标准，数据极度稳定，包含丰富的基础数据、因子数据甚至资金流向数据。只需要注册并获取一定积分即可免费调用大部分 A 股日线及基本面数据。
  - **建议**：用 Tushare Pro 替换 Baostock 作为主力历史数据源。
- **备选 (完全免费但需工程优化)：AkShare + 强力重试机制**
  - **优势**：完全开源免费，更新极快。
  - **建议**：在代码中封装带有指数退避（Exponential Backoff）的重试装饰器，并限制并发请求数。

### 2. 核心范式转变：从“即时下载”到“离线数据湖”
- **彻底剥离下载与训练**：不再在 `runner.py` 中下载数据。编写一个独立的 `scripts/sync_daily_data.py`。
- **每日定时增量同步**：在每天收盘后（如 18:00），运行同步脚本，将当天的最新数据拉取到本地，实现“一次下载，永久本地使用”。

## 三、 数据存储与加载架构升级

为了实现极速的数据读取，建议采用 **“Parquet + DuckDB/Pandas”** 的本地数据湖架构。

### 1. 存储结构设计 (Local Data Lake)
废弃之前的按请求范围缓存的方式，改为**按股票代码/年份进行独立存储**：
```text
data/
└── local_lake/
    ├── daily_k/                 # 日线量价数据
    │   ├── sh.600000.parquet    # 包含该股票上市至今的所有日线数据
    │   ├── sz.000001.parquet
    │   └── ...
    ├── basics/                  # 基本面数据 (市值、财务指标等)
    └── features/                # 预计算好的特征缓存
        └── v1_macd_rsi/
            └── sh.600000.parquet
```

### 2. 读取与加载引擎优化
- **全量加载 + 内存切片 (Memory Slicing)**：
  - 一只股票 20 年的日线数据（约 5000 行）存储为 Parquet 只有几百 KB。当 `DataRepository` 请求数据时，直接 `pd.read_parquet('sh.600000.parquet')` 加载全量历史，然后在内存中通过 `df.loc[start:end]` 进行切片返回。这种方式比网络请求快上万倍。
- **引入 DuckDB 加速 (可选进阶)**：
  - 如果未来要进行横截面分析（比如同时读取 5000 只股票在某一天的数据），Pandas 循环读取单个文件会变慢。此时可以引入 `DuckDB`，通过 SQL 原生查询整个 `daily_k/*.parquet` 目录，速度极快。
- **特征数据缓存 (Feature Caching)**：
  - 针对算好的因子（如 MACD, RSI, Label 等），根据配置的 Hash 值将其存入 `features/<hash>/` 目录。训练时直接 Load 特征表，免去重复计算。

## 四、 实施路线图 (Action Plan)

如果您决定动手，我们将按照以下顺序实施：

1. **第一步：编写离线同步脚本**
   - 开发 `scripts/sync_data.py`。
   - 实现全量下载逻辑（初始化用）和增量更新逻辑（每日收盘用）。
   - 确立以 `symbol.parquet` 为单位的存储格式。
2. **第二步：改造 DataRepository**
   - 重写 `get_daily_data` 方法。移除原来的网络请求逻辑。
   - 改为：检查本地 `data/local_lake/daily_k/{symbol}.parquet` 是否存在，存在则读取全量并进行时间范围过滤；不存在则报错提示运行同步脚本。
3. **第三步：(可选) 引入 Tushare 接口**
   - 在 `data/source/` 下新增 `tushare_source.py`，作为更稳定的数据下载器。
4. **第四步：引入特征缓存机制**
   - 在 `FeaturePipeline` 中增加检测机制，将转换后的 `features_df` 持久化，进一步提速。

这种**“离线数据同步 + 本地 Parquet 内存切片”**的架构是目前专业量化机构（在不使用 DolphinDB 等昂贵时序数据库的情况下）最通用的高性能解决方案。
